# V2.3-C Phase 2 — C6 (Override Approved Prices) + C8 (Hard-Delete Customer) — Design Spec

**Status:** **ASSUMED DEFAULTS** — drafted from brainstorm proposed positions; user can override D-decisions before plan phase.

## 1. Motivation

After V2.3-C Phase 1 shipped corrective ops (delete, rollback, reassign-SP, unlock-bom, unlock-costing, reset-password, audit-log), two operations remained deferred:

- **C6 — Override approved prices.** Real-world need: market price moved between MD approval and customer signoff; admin needs to issue revised prices without breaking history.
- **C8 — Hard-delete customer.** Real-world need: GDPR-style erasure or duplicate cleanup; current API has no "remove customer" path.

P2 closes both, plus carries the P1-deferred technical debt (controller split, N+1 notification fan-out, audit-log filter UI gaps).

- **Web UI:** `OverridePricesModal` + `DeleteCustomerModal`; wire `AdminActionsCard` (Approved status only) and `CustomersPage` row action
- **Web UI:** add `adminUserId` + `entityId` filter inputs to `/admin/audit-log` page

- Mobile UI (D8 carry-forward — web only, consistent with P1)
- Schema change to nullable `QuotationRequest.CustomerId` (C8 uses anonymize-in-place pattern that preserves FK integrity)
- "Restore deleted customer" endpoint (D13 — one-way operation)
- Reverting a C6 override via dedicated endpoint (a second C6 with old values *is* the revert; D6)
- Replacing JSON-pre DiffPanel with real JSON diff (separate P2 cleanup task, not blocking)

## 5. Endpoint contracts

### 5.1 `POST /api/admin/requisitions/{id}/override-prices` (C6)

**Request body:**
```jsonc
{
  "reason": "string (required, ≥ 5 chars)",
  "items": [
    {
      "requisitionItemId": 1,
      "salesPricePerKgAed": 12.5,
      "salesPricePerKgForeign": null,            // optional, presence required if currency != AED
      "profitMarginPct": 22.0,
      "materialCostPct": 65.0,
      "otherCostPct": 13.0
    }
    // … one entry per existing RequisitionItem (D5: item set frozen)
  ]
  // No sendEmail flag — D6 broad policy: app always emails SP (never customer) if SP has email
}
```

**Validation:**
- 401 / 403 if not Admin
- 400 if req.Status != `Approved`
- 400 if `reason` < 5 chars
- 400 if `items` is missing any existing `RequisitionItem.Id` (item set must be exhaustive — D5)
- 400 if `items` contains a `requisitionItemId` not on this req
- 400 if any `*PricePerKg*` < 0
- 400 if `profitMarginPct + materialCostPct + otherCostPct` doesn't sum to ~100% (±0.01 tolerance — same as MD-approve validation)
- 400 if currency != AED and any `salesPricePerKgForeign` is null

**Side effects (all in one transaction):**
1. Capture `BeforeJson` snapshot of current `QuotationApproval` (ApprovedAt, ApprovedByUserId, IsApproved, Items[]).
2. Mark current `QuotationApproval.IsSuperseded = true; SupersededAt = now`.
3. Capture current exchange rate (`Currency.Code != "AED"` only) via `ExchangeRateService.GetCurrentRateAsync(req.CurrencyCode)` — or null if AED.
4. Insert new `QuotationApproval` with `IsApproved = true; ApprovedAt = now; ApprovedByUserId = adminUserId; Notes = "[Override] " + reason`. New `ApprovalItem` rows with payload prices.
5. Generate fresh PDF via `PdfService` and send email via `EmailService` to **SalesPerson** (D6 broad policy — customer is never the email recipient). Email body must include `customer.Name`, `customer.Email`, `customer.PhoneNumber` so SP can forward to customer through their own channel. If SP has no email on record, log a warning and skip the email step (don't fail the override).
6. Persist 3 notif rows + SignalR push: SP, original-MD, Accountants in branch.
7. `AdminAuditLogger.Log(OverridePrices, BeforeJson=..., AfterJson=...)`.

**Response:**
- 200 OK with `{ newApprovalId, supersededApprovalId, emailSentToSpUserId: int? }` (null if SP had no email)

### 5.2 `DELETE /api/admin/customers/{id}` (C8)

**Request body:**
```json
{
  "reason": "string (required, ≥ 5 chars)"
}
```

**Validation:**
- 401 / 403 if not Admin
- 404 if customer not found
- 409 Conflict if customer is already soft-deleted (`IsDeleted == true`)
- 409 Conflict if customer has any req in active workflow status (D13). Response body lists offending req-IDs:
  ```json
  { "error": "Customer has in-flight requisitions. Resolve them first.", "blockingRequisitions": [123, 456] }
  ```
- 400 if `reason` < 5 chars

**Side effects (one transaction):**
1. Capture `BeforeJson` of full Customer row + array of all referencing `RequisitionRequest.Id`s (no full req payloads — they're in DB).
2. Set `Customer.IsDeleted = true; DeletedAt = now; DeletedByUserId = adminUserId`.
3. Anonymize fields: `Name = $"[Deleted {DateOnly.FromDateTime(DateTime.UtcNow):yyyy-MM-dd}]"`, `Code = ""`, `Email = ""`, `PhoneNumber = ""`, `Address = ""`, `SalesPersonId = null`.
4. Persist notif rows + SignalR push: customer's `SalesPersonId` user (pre-anonymization), their V23b group peers, Accountants associated with customer's reqs' branches.
5. `AdminAuditLogger.Log(HardDeleteCustomer, BeforeJson=..., AfterJson=anonymized row)`.

**Response:** 204 No Content.

**Listing impact:**
- `GET /api/customers` and `GET /api/customers/{id}` filter on `c => !c.IsDeleted` (default).
- `RequisitionDetail` (and historical reads) keep showing the anonymized customer (it still resolves via FK). Display logic shows `[Deleted YYYY-MM-DD]` as the customer name and an italic helper "Customer was deleted on YYYY-MM-DD".

## 7. Refactor: `NotificationService.SendToUsersAsync` (D19)

Add new batched method:

```csharp
public async Task SendToUsersAsync(
    IEnumerable<int> userIds,
    NotificationType type,
    string message,
    int? quotationRequestId = null,
    int? customerId = null,
    CancellationToken ct = default)
{
    var rows = userIds.Distinct().Select(uid => new Notification {
        UserId = uid,
        Type = type,
        Message = message,
        QuotationRequestId = quotationRequestId,
        CustomerId = customerId,
        CreatedAt = DateTime.UtcNow,
        IsRead = false
    }).ToList();
    db.Notifications.AddRange(rows);
    await db.SaveChangesAsync(ct);
    foreach (var uid in userIds.Distinct())
        await hub.Clients.User(uid.ToString()).SendAsync("notification", new {/* … */}, ct);
}
```

Migrate all 7 P1 endpoints + 2 P2 endpoints + any existing N+1 caller in `BomController` / `CostingController` / `RequisitionsController` / `ApprovalsController` to use the batched form.

This closes the **carry-forward N+1 debt** flagged in P1 lesson 10.

## 10. Notification routing details

### C6 (PricesOverridden)
Recipients computed in endpoint:
```csharp
var recipients = new HashSet<int>();
recipients.Add(req.SalesPersonId);
recipients.Add(currentApproval.ApprovedByUserId); // original MD
recipients.UnionWith(
    db.Users.Where(u => u.Role == UserRole.Accountant && u.IsActive)
        .Where(u => db.UserBranches.Any(ub => ub.UserId == u.Id && ub.BranchId == req.BranchId))
        .Select(u => u.Id));
recipients.Remove(currentUserId); // don't notify self (admin)
await notify.SendToUsersAsync(recipients, NotificationType.PricesOverridden, ...);
```

### C8 (CustomerDeleted)
```csharp
var recipients = new HashSet<int>();
if (customer.SalesPersonId.HasValue)
{
    recipients.Add(customer.SalesPersonId.Value);
    var spGroup = db.Users.Where(u => u.Id == customer.SalesPersonId.Value).Select(u => u.GroupId).FirstOrDefault();
    if (spGroup.HasValue)
    {
        recipients.UnionWith(
            db.Users.Where(u => u.Role == UserRole.SalesPerson && u.GroupId == spGroup && u.IsActive)
                .Select(u => u.Id));
    }
}
var customerReqBranches = db.QuotationRequests.Where(r => r.CustomerId == customer.Id).Select(r => r.BranchId).Distinct().ToList();
recipients.UnionWith(
    db.Users.Where(u => u.Role == UserRole.Accountant && u.IsActive)
        .Where(u => db.UserBranches.Any(ub => ub.UserId == u.Id && customerReqBranches.Contains(ub.BranchId)))
        .Select(u => u.Id));
recipients.Remove(currentUserId);
await notify.SendToUsersAsync(recipients, NotificationType.CustomerDeleted, ...);
```
